# can_driver 配置文件字段详解与从零配置指南

本文档专门解释 `can_driver/config/` 下两份配置文件的字段含义，并给出一套从零落地的配置步骤。

覆盖范围：
- `config/can_driver.yaml`
- `config/ros_controllers.yaml`
- `launch/can_driver.launch` 对这两份文件的加载方式

不覆盖的内容：
- 电机 CAN 私有协议细节
- MoveIt 控制器映射细节
- UI 调试工具的单独用法

## 1. 配置是如何生效的

`can_driver.launch` 的加载顺序是：

1. 加载 `config/can_driver.yaml`
2. 如果 `use_controllers:=true`，再加载 `config/ros_controllers.yaml`
3. 启动 `can_driver_node`
4. 如果 `use_controllers:=true`，再由 `controller_manager/spawner` 启动控制器

其中：
- `can_driver.yaml` 里的参数挂在 `/can_driver_node/*` 私有命名空间下
- `ros_controllers.yaml` 里的参数由 `controller_manager` 和各控制器插件读取

可以把这两份文件理解成两层：
- `can_driver.yaml` 负责“硬件层”
  - 哪些 joint 归哪个电机
  - 电机走哪个 CAN 口
  - 电机讲哪种协议
  - 这个 joint 暴露成位置接口还是速度接口
- `ros_controllers.yaml` 负责“控制器层”
  - 启什么控制器
  - 控制器接管哪些 joint
  - 控制器更新频率、里程计参数、轨迹约束等

## 2. `config/can_driver.yaml` 详解

### 2.1 文件结构

文件根节点是：

```yaml
can_driver_node:
  control_frequency: 100.0
  ...
  joints:
    - name: elbow_pitch_joint
      ...
```

也就是说，代码真正读取的是：
- `/can_driver_node/control_frequency`
- `/can_driver_node/motor_query_hz`
- `/can_driver_node/joints`

如果你把这些字段写在 `can_driver_node:` 之外，节点是读不到的。

### 2.2 节点级参数

下表中的“代码默认值”指的是：即使 YAML 里不写，`CanDriverHW.cpp` 也会回退到这个值。

| 字段 | 类型 | 代码默认值 | 作用 | 什么时候需要改 |
| --- | --- | --- | --- | --- |
| `control_frequency` | `double` | `100.0` | 主控制循环频率。每个周期执行 `read -> controller_manager.update -> write`。 | 总线负载高、控制器频率要统一、或需要更高控制带宽时。 |
| `direct_cmd_timeout_sec` | `double` | `0.5` | 直通 topic 命令的有效时长。超过时间后，旧的 `~/motor/<joint>/cmd_position` 或 `cmd_velocity` 不再继续当作新命令使用。 | 用 topic 直控电机时。 |
| `motor_state_period_sec` | `double` | `0.1` | `~/motor_states` 的发布周期，单位秒。 | 需要更快或更慢的状态发布时。 |
| `motor_query_hz` | `double` | `0.0` | 电机反馈查询频率。`>0` 表示固定频率轮询，`<=0` 表示交给协议层默认策略。 | 想显式控制反馈刷新频率时。 |
| `direct_cmd_queue_size` | `int` | `1` | 直通命令 topic 订阅队列长度。 | 高频手动调试时。 |
| `debug_bypass_ros_control` | `bool` | `false` | 允许直通 topic 命令绕过 ros_control 的回退路径，便于单轴联调。 | 只建议在单轴调试时打开。 |
| `startup_position_sync_timeout_sec` | `double` | `1.0` | 生命周期 `init` 期间等待首轮真实反馈、并把位置命令对齐到当前反馈的超时时间。 | 首次反馈较慢或总线负载较大时。 |
| `startup_probe_query_hz` | `double` | `5.0` | 初始化阶段临时使用的探测查询频率。代码会在启动同步阶段把设备刷新率暂时切到这个值。 | 启动时想更快拿到首帧反馈时。 |
| `pp_fast_write_enabled` | `bool` | `false` | PP 协议位置/速度命令是否走快写。`true` 更激进，`false` 更保守。 | PP 电机调优时。 |
| `pp_position_default_velocity_rad_s` | `double` | 约 `1.0472` | PP `position` 路径的默认目标速度，单位 `rad/s`。代码会按每个 joint 的 `velocity_scale` 换算成协议原始值。 | PP 位置模式的预配置速度要更快或更慢时。 |
| `pp_csp_default_velocity_rad_s` | `double` | 约 `1.0472` | PP `csp` 路径的默认目标速度，单位 `rad/s`。同样按每轴 `velocity_scale` 换算。 | CSP 轴初始化或运行前置速度要调整时。 |
| `safety_stop_on_fault` | `bool` | `true` | 发现故障反馈时是否自动发一次 Stop，并阻断后续运动命令。 | 一般不建议关闭。 |
| `safety_require_enabled_for_motion` | `bool` | `true` | 下发运动命令前是否要求轴处于 enabled。 | 一般不建议关闭。 |
| `max_position_step_rad` | `double` | `0.0` | 单周期位置步进上限，单位 `rad`。`<=0` 表示关闭。超过后会被截断并告警。 | 想限制单拍跳变时。 |
| `safety_hold_after_device_recover` | `bool` | `true` | 掉线恢复后是否先 hold 一个控制周期，并清空直通命令，避免链路恢复瞬间误动作。 | 一般建议保持开启。 |
| `pp_local_zero_offset_persistence_enabled` | `bool` | `false` | 是否把 PP 零点偏置持久化到本机文件，并在下次初始化时恢复。 | 想把零点保存在驱动侧时。 |
| `pp_zero_offset_persist_on_apply_to_motor` | `bool` | `true` | 当 `apply_to_motor=true` 时，是否继续向电机发持久化命令，把零点写到电机非易失存储。 | 不想改电机 EEPROM 时可关掉。 |
| `pp_local_zero_offset_file` | `string` | `$HOME/.ros/can_driver_pp_zero_offsets.txt` | 本地零点偏置持久化文件路径。 | 想自定义零点文件位置时。 |

几点补充：

- `control_frequency` 最好和 `ros_controllers.yaml` 里的 `controller_manager.update_rate` 保持一致。
- `pp_position_default_velocity_rad_s` 和 `pp_csp_default_velocity_rad_s` 是“角速度”，不是旧文档里提过的 raw 值。
- `pp_local_zero_offset_file` 的文本格式很简单：每行一个 `motor_id zero_offset_rad`。
- `debug_bypass_ros_control` 只影响 `~/motor/<joint>/cmd_*` 直通链路，不会改变 controller plugin 的行为。

### 2.3 一个容易踩坑的历史注释

你会在旧注释里看到 `init_loopback` 这个名字，但当前代码并不会从 YAML 读取这个字段。

当前真实生效的 loopback 开关来自：
- `~/init` 服务请求里的 `loopback` 字段

也就是说：
- 改 `can_driver.yaml` 里的 `init_loopback` 不会生效
- 如果你想让当前 socket 接收自己发出的帧，要在调用 `~/init` 时显式传 `loopback: true`

### 2.4 `joints` 数组的校验规则

`joints` 是整份文件最关键的部分。它定义“逻辑关节”和“物理电机”的映射关系。

代码会做这些硬校验：
- `joints` 必须是非空数组
- 每个 joint 都必须有 `name / motor_id / protocol / can_device / control_mode`
- `name` 不能重复
- `motor_id` 不能重复
- `(can_device, protocol, on-wire node id)` 组合不能冲突
- `position_scale / velocity_scale` 必须是正数
- `direction_sign` 只能是 `1.0` 或 `-1.0`
- `ip_max_velocity / ip_max_acceleration / ip_max_jerk` 必须大于 `0`
- `ip_goal_tolerance` 必须大于等于 `0`

### 2.5 `joints[*]` 每个字段的含义

| 字段 | 是否必填 | 说明 | 约束 / 建议 |
| --- | --- | --- | --- |
| `name` | 必填 | 逻辑关节名。用于对接 URDF、`joint_states`、controller、MoveIt。 | 必须和 URDF 里的 joint 名完全一致。 |
| `motor_id` | 必填 | 电机 ID。支持十进制整数，也支持字符串形式的十六进制，例如 `0x13`。 | 取值范围 `0~65535`，全系统唯一。 |
| `protocol` | 必填 | 电机协议类型。当前支持 `MT` 和 `PP`。 | `PP` 常用于机械臂轴，`MT` 常用于底盘速度轴。 |
| `can_device` | 必填 | 这个电机挂在哪个 SocketCAN 设备上，例如 `can0`、`can1`、`vcan0`。 | 必须是当前机器上真实存在并已 `up` 的接口名。 |
| `control_mode` | 必填 | 关节暴露给 ros_control 的控制语义。支持 `velocity`、`position`、`csp`。 | `velocity` 会注册 `VelocityJointInterface`；`position` 和 `csp` 都注册 `PositionJointInterface`。 |
| `position_scale` | 可选 | 原始位置反馈到 `rad` 的换算比例。 | 可直接填比例，也可填 PPR 整数。若填 `65536`，代码会自动换成 `2π/65536`。 |
| `velocity_scale` | 可选 | 原始速度反馈到 `rad/s` 的换算比例。 | 与 `position_scale` 规则相同。 |
| `direction_sign` | 可选 | 方向修正系数。 | 只能填 `1.0` 或 `-1.0`。会同时作用到位置和速度换算。 |
| `ip_max_velocity` | 可选 | 预留给 IP 轨迹执行器的每轴最大速度。 | 当前 `can_driver` 主控制循环不会直接消费它，但 parser 会校验并保存。 |
| `ip_max_acceleration` | 可选 | 预留给 IP 轨迹执行器的每轴最大加速度。 | 当前主控制循环不会直接消费它。 |
| `ip_max_jerk` | 可选 | 预留给 IP 轨迹执行器的每轴最大 jerk。 | 当前主控制循环不会直接消费它。 |
| `ip_goal_tolerance` | 可选 | 预留给 IP 轨迹执行器的终点容差。 | 当前主控制循环不会直接消费它。 |

### 2.6 `control_mode` 到接口语义的对应关系

这是从零配置时最容易配错的一项。

| `control_mode` | 暴露给 ros_control 的接口 | 适合接哪个控制器 |
| --- | --- | --- |
| `velocity` | `VelocityJointInterface` | `diff_drive_controller`、速度型单轴控制器 |
| `position` | `PositionJointInterface` | `JointTrajectoryController`、位置型单轴控制器 |
| `csp` | `PositionJointInterface` | `JointTrajectoryController`，但底层在初始化时会切到电机 CSP 模式 |

几点建议：

- `velocity` 常给底盘轮、履带、连续转动执行器。
- `position` 常给普通位置轴。
- `csp` 从当前代码路径看是围绕 PP 协议轴设计的；从零配置时，不要把 `MT` 轴写成 `csp`。

### 2.7 `position_scale / velocity_scale` 怎么填

这两个字段支持两种写法。

第一种：直接填换算系数。

```yaml
position_scale: 9.587379924285257e-05
velocity_scale: 9.587379924285257e-05
```

含义是：

```text
物理量 = raw * scale
```

第二种：直接填每圈脉冲数 PPR。

```yaml
position_scale: 65536
velocity_scale: 65536
```

代码会自动把它解释为：

```text
scale = 2π / PPR
```

如果你的编码器确实是每圈 65536 count，这种写法通常最直观。

### 2.8 一个最小 joint 示例

PP 协议、CSP 机械臂轴：

```yaml
can_driver_node:
  joints:
    - name: elbow_pitch_joint
      motor_id: 0x13
      protocol: PP
      can_device: can1
      control_mode: csp
      position_scale: 65536
      velocity_scale: 65536
      direction_sign: -1.0
```

MT 协议、速度底盘轴：

```yaml
can_driver_node:
  joints:
    - name: left_wheel
      motor_id: 321
      protocol: MT
      can_device: can0
      control_mode: velocity
      position_scale: 65536
      velocity_scale: 65536
```

## 3. `config/ros_controllers.yaml` 详解

### 3.1 这份文件什么时候生效

只有在：

```bash
roslaunch can_driver can_driver.launch use_controllers:=true
```

时，`ros_controllers.yaml` 才会被加载。

如果 `use_controllers:=false`：
- `can_driver_node` 仍然会启动
- 但控制器参数不会加载
- `controller_manager/spawner` 也不会启动控制器
- 这更接近“直通调试模式”

即使 `use_controllers:=true`，你也还要区分两件事：
- 控制器参数是否已经加载
- 对应控制器是否真的被 spawner 拉起来

当前仓库里的 `launch/can_driver.launch` 默认只会启动：
- `joint_state_controller`

而 `arm_controller`、`wheel_controller` 仍在注释块里，所以从零接入时不要只改 YAML，还要同步检查 launch 里的 spawner 配置。

### 3.2 `controller_manager`

```yaml
controller_manager:
  update_rate: 100
```

| 字段 | 说明 | 建议 |
| --- | --- | --- |
| `update_rate` | `controller_manager` 的更新频率。 | 尽量与 `can_driver.yaml` 中的 `control_frequency` 保持一致。 |

### 3.3 `joint_state_controller`

```yaml
joint_state_controller:
  type: joint_state_controller/JointStateController
  publish_rate: 50
```

| 字段 | 说明 |
| --- | --- |
| `type` | 控制器插件类型。这个控制器只发布状态，不下发运动命令。 |
| `publish_rate` | `/joint_states` 发布频率。 |

这通常是第一批就应该启动的控制器，因为：
- RViz 要靠它显示关节状态
- `robot_state_publisher` 要靠它算 TF
- MoveIt 也依赖它获取当前机器人状态

### 3.4 `arm_controller`

```yaml
arm_controller:
  type: position_controllers/JointTrajectoryController
  joints:
    - elbow_pitch_joint
```

| 字段 | 说明 | 约束 |
| --- | --- | --- |
| `type` | 轨迹控制器插件类型。当前文件使用 `position_controllers/JointTrajectoryController`。 | 适合 `position/csp` 语义关节。 |
| `joints` | 这个轨迹控制器要接管的关节列表。 | 每个 joint 必须已经存在于 `can_driver.yaml`，且名称与 URDF 完全一致。 |
| `constraints.goal_time` | 可选。目标允许超时窗口。 | 轨迹收尾容忍度调优时使用。 |
| `constraints.stopped_velocity_tolerance` | 可选。到终点时速度容差。 | 收尾抖动或误判到位时调。 |

注意：
- `arm_controller.joints` 可以是 `joints` 总表的子集。
- 但这些关节必须是位置语义，也就是 `control_mode=position` 或 `csp`。
- 如果你把 `velocity` 关节塞进这个控制器，语义就错了。

### 3.5 `wheel_controller`

```yaml
wheel_controller:
  type: diff_drive_controller/DiffDriveController
  left_wheel: left_wheel
  right_wheel: right_wheel
  wheel_separation: 0.5
  wheel_radius: 0.1
```

| 字段 | 说明 | 约束 |
| --- | --- | --- |
| `type` | 差速底盘控制器插件。 | 适合两侧轮速控制。 |
| `left_wheel` | 左轮 joint 名。 | 必须存在于 `can_driver.yaml`，且通常应为 `velocity` 语义。 |
| `right_wheel` | 右轮 joint 名。 | 同上。 |
| `wheel_separation` | 左右轮中心距，单位米。 | 必须按实车测量值填写。 |
| `wheel_radius` | 轮半径，单位米。 | 必须按实车测量值填写。 |
| `publish_rate` | 可选，`/odom` 发布频率。 | 需要更细或更粗的里程计频率时填写。 |
| `odom_frame_id` | 可选，里程计坐标系名。 | 默认通常是 `odom`。 |
| `base_frame_id` | 可选，机器人本体坐标系名。 | 默认通常是 `base_link`。 |
| `linear.x.max_velocity` | 可选，线速度上限。 | 建议填，保护底盘。 |
| `linear.x.max_acceleration` | 可选，线加速度上限。 | 建议填。 |
| `angular.z.max_velocity` | 可选，角速度上限。 | 建议填。 |
| `angular.z.max_acceleration` | 可选，角加速度上限。 | 建议填。 |

### 3.6 控制器文件最常见的三类错误

1. joint 名写对了控制器，却没写进 `can_driver.yaml`
2. joint 在 `can_driver.yaml` 里是 `velocity`，却被塞进位置轨迹控制器
3. `wheel_separation` 和 `wheel_radius` 没按实物填写，导致里程计系统性偏差

## 4. 从零配置指南

这一节假设你要第一次把一套电机接到 `can_driver` 上。

### 第 1 步：先画一张电机映射表

在动 YAML 之前，先把这些信息整理成表：

| joint 名 | 电机 ID | 协议 | CAN 口 | 控制语义 |
| --- | --- | --- | --- | --- |
| `elbow_pitch_joint` | `0x13` | `PP` | `can1` | `csp` |
| `left_wheel` | `321` | `MT` | `can0` | `velocity` |

这里最重要的是统一三件事：
- `joint name`
- `motor_id`
- `control_mode`

后面所有文件都会用到它们。

### 第 2 步：先把 URDF 准备好

从零配置时，很多人先改 YAML，最后才补 URDF。这样往往会在启动后才发现：
- `joint_state_controller` 有名字，但 RViz 不显示
- 限位没有被加载
- controller 能启动，但 MoveIt 不认

所以建议你先保证：
- `robot_description` 能正常加载
- 每个要控制的 joint 都真的在 URDF 里
- 位置轴在 URDF 里有基本限位

`can_driver` 的限位读取顺序是：
1. 先读 `robot_description` 里的 URDF joint limits
2. 再读 rosparam 里的 joint limits 覆盖项

也就是说：
- 限位主来源不是 `can_driver.yaml`
- `can_driver.yaml` 负责电机映射，不负责常规 joint 限位

### 第 3 步：写一份最小可启动的 `can_driver.yaml`

建议从最小配置开始，不要一上来把全部高级参数都打开。

```yaml
can_driver_node:
  control_frequency: 100.0
  motor_query_hz: 20.0
  startup_position_sync_timeout_sec: 1.0
  safety_stop_on_fault: true
  safety_require_enabled_for_motion: true

  joints:
    - name: elbow_pitch_joint
      motor_id: 0x13
      protocol: PP
      can_device: can1
      control_mode: csp
      position_scale: 65536
      velocity_scale: 65536
      direction_sign: -1.0
```

第一轮建议只保留：
- `control_frequency`
- `motor_query_hz`
- `startup_position_sync_timeout_sec`
- 两个 safety 开关
- 至少一个 joint

其余高级项可以在跑通之后再加。

### 第 4 步：逐个补 joint，而不是一次性全贴进去

推荐顺序：

1. 先只配 1 个最容易确认方向和限位的 joint
2. 启动、验证这个 joint 的反馈和命令都正常
3. 再逐个往 `joints:` 里追加

这样做的原因是：
- `name` 冲突更容易定位
- `motor_id` 冲突更容易定位
- 某个轴的 `direction_sign` 或 scale 写错时，不会把整套系统搅乱

### 第 5 步：按控制语义写 `ros_controllers.yaml`

如果你先只想验证机械臂轴：

```yaml
controller_manager:
  update_rate: 100

joint_state_controller:
  type: joint_state_controller/JointStateController
  publish_rate: 50

arm_controller:
  type: position_controllers/JointTrajectoryController
  joints:
    - elbow_pitch_joint
```

如果你要验证底盘：

```yaml
controller_manager:
  update_rate: 100

joint_state_controller:
  type: joint_state_controller/JointStateController
  publish_rate: 50

wheel_controller:
  type: diff_drive_controller/DiffDriveController
  left_wheel: left_wheel
  right_wheel: right_wheel
  wheel_separation: 0.42
  wheel_radius: 0.082
```

记住两条硬规则：
- 位置控制器只接 `position/csp` 语义 joint
- 差速控制器只接 `velocity` 语义 wheel joint

### 第 6 步：启动时先区分“直通调试”还是“控制器模式”

如果你只想先看节点和状态：

```bash
roslaunch can_driver can_driver.launch use_controllers:=false
```

如果你要让 controller_manager 真正接管：

```bash
roslaunch can_driver can_driver.launch use_controllers:=true
```

这一步经常被忽略。很多“为什么 MoveIt 规划了但电机不动”的问题，根因就是 `use_controllers` 没开，或者 spawner 只起了 `joint_state_controller`。

如果你已经写好了 `arm_controller` 或 `wheel_controller`，但 `list_controllers` 里还是看不到它们的 `running` 状态，优先检查：
- `launch/can_driver.launch` 里是否把对应 controller 名加入了 spawner `args`
- 是否仍然停留在当前默认的“只起 joint_state_controller”状态

### 第 7 步：做一次最小生命周期启动

以 `can1` 上的一组关节为例：

```bash
rosservice call /can_driver_node/init "{device: 'can1', loopback: false}"
rosservice call /can_driver_node/enable "{}"
rosservice call /can_driver_node/resume "{}"
```

如果你在 `vcan` 或本机回环测试：

```bash
rosservice call /can_driver_node/init "{device: 'vcan0', loopback: true}"
```

注意：
- 当前 loopback 是 `~/init` 请求参数，不是 YAML 参数。
- 从零联调时，建议先在单个 CAN 口、单组电机上跑通。

### 第 8 步：按下面的顺序验收

推荐检查顺序：

1. `rostopic echo /joint_states`
2. `rostopic echo /can_driver_node/motor_states`
3. `rostopic echo /can_driver_node/lifecycle_state`
4. `rosservice call /controller_manager/list_controllers`
5. 再去发控制命令

如果第 1 步就没有数据，先别去怀疑控制器，先查：
- CAN 口是否存在且 `up`
- `motor_id / protocol / can_device` 是否写对
- `~/init` 是否成功

如果第 1 步有数据但第 4 步没有 `running` 控制器，再查：
- `use_controllers` 是否为 `true`
- `ros_controllers.yaml` 是否被加载
- spawner 是否真的起了你需要的控制器

### 第 9 步：最后再调这几类高级参数

等最小系统跑通后，再考虑调：
- `pp_fast_write_enabled`
- `pp_position_default_velocity_rad_s`
- `pp_csp_default_velocity_rad_s`
- `max_position_step_rad`
- `pp_local_zero_offset_persistence_enabled`

这样能把“系统没起来”和“性能没调好”这两类问题分开。

## 5. 一个推荐的最小落地顺序

如果你现在要新接一套电机，最稳妥的顺序是：

1. 先在 URDF 里把 joint 名和限位写对
2. 再写 `can_driver.yaml` 的 1 个 joint 最小配置
3. 跑通 `init -> enable -> resume`
4. 看 `motor_states` 和 `joint_states`
5. 再补 `ros_controllers.yaml`
6. 再让 MoveIt / Navigation 接进来
7. 最后才调快写、默认速度、零点持久化这类高级项

## 6. 速查清单

### 必须一致的名字

- `can_driver.yaml -> joints[*].name`
- URDF 里的 `<joint name="...">`
- `ros_controllers.yaml` 里的关节名
- MoveIt 配置里的关节名

### 最容易配错的字段

- `control_mode`
- `direction_sign`
- `position_scale / velocity_scale`
- `wheel_separation / wheel_radius`

### 当前代码支持、但示例里可能没写出来的字段

- `direct_cmd_timeout_sec`
- `motor_state_period_sec`
- `direct_cmd_queue_size`
- `startup_probe_query_hz`
- `pp_local_zero_offset_file`

### 当前不要再按旧文档去写的字段

- `pp_position_default_velocity_raw`
- `pp_csp_default_velocity_raw`
- `init_loopback`

这些名字不是当前主代码实际读取的配置口径。
