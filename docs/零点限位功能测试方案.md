# 零点限位功能测试方案

目标：验证 `set_zero_limit` 接口在“零点偏置 + URDF限位 + 电机物理限位 + 上电位置匹配”链路上可用且安全。

## 1. 测试范围

- 服务接口：`/can_driver_node/set_zero_limit`
- 软件限位：`CanDriverHW::write()` 中命令钳制
- 物理限位下发（PP）：
  - `0x3B`（零点偏置）
  - `0x39`（上限）
  - `0x3A`（下限）
  - `0x38`（限位使能）
- 启动一致性：上电读取当前位置并与限位匹配（超限拒绝激活）

## 1.1 一键脚本（推荐）

已提供一键测试脚本：

- [scripts/zero_limit_one_click_test.sh](../scripts/zero_limit_one_click_test.sh)

运行方式：

```bash
source /home/fire/catkin_ws/devel/setup.bash
bash /home/fire/catkin_ws/src/can_driver/scripts/zero_limit_one_click_test.sh
```

可选环境变量：

- `NS`（默认 `/can_driver_node`）
- `MOTOR_ID`（默认 `22`）
- `CAN_DEV`（默认 `can0`）
- `JOINT_NAME`（默认 `rotary_table`）

## 2. 前置条件

1. `can_driver_node` 可正常启动。
2. 被测关节为 `control_mode: position`，建议先用 `rotary_table`（`motor_id=22`）。
3. 已有 `robot_description`（若测试 `use_urdf_limits=true`）。
4. 可抓包：`candump can0`。

## 3. 测试用例

### T1 接口可用性（基础）

步骤：
1. 查看服务：
   - `rosservice list | grep set_zero_limit`
2. 查看接口定义：
   - `rossrv show can_driver/SetZeroLimit`

预期：
- 服务存在，字段与设计一致。

---

### T2 手工限位设置 + 物理下发（PP）

步骤：
1. 开抓包：
   - `candump can0`
2. 调用：
   - `rosservice call /can_driver_node/set_zero_limit "motor_id: 22
zero_offset_rad: 0.0
min_position_rad: -1.0
max_position_rad: 1.0
use_urdf_limits: false
apply_to_motor: true"`

预期：
- 服务 `success=true`。
- 返回 `applied_min_rad=-1.0`、`applied_max_rad=1.0`。
- 抓包可见 PP 写寄存器帧：`0x3B/0x39/0x3A/0x38`。

---

### T3 仅软件生效（不下发电机）

步骤：
1. 调用：
   - `rosservice call /can_driver_node/set_zero_limit "motor_id: 22
zero_offset_rad: 0.1
min_position_rad: -1.0
max_position_rad: 1.0
use_urdf_limits: false
apply_to_motor: false"`
2. 发送越界位置命令：
   - `rostopic pub -1 /can_driver_node/motor/rotary_table/cmd_position std_msgs/Float64 "data: 2.0"`

预期：
- 服务成功。
- 抓包不应出现 `0x3B/0x39/0x3A/0x38` 设置帧。
- 实际下发位置目标被钳制到软件限位范围内（结合 `motor_states` 验证）。

---

### T4 URDF限位读取 + 偏置叠加

步骤：
1. 确认 URDF 有该关节位置限位。
2. 调用：
   - `rosservice call /can_driver_node/set_zero_limit "motor_id: 22
zero_offset_rad: 0.2
min_position_rad: 0.0
max_position_rad: 0.0
use_urdf_limits: true
apply_to_motor: false"`

预期：
- 服务成功。
- 返回的 `applied_min_rad/applied_max_rad` 等于 URDF 原始限位整体平移 `+0.2rad`。

---

### T5 当前位置匹配拦截（安全）

步骤：
1. 先将电机移动到某位置（如约 `+1.0rad`）。
2. 设置不包含当前位置的限位：
   - `rosservice call /can_driver_node/set_zero_limit "motor_id: 22
zero_offset_rad: 0.0
min_position_rad: -0.2
max_position_rad: 0.2
use_urdf_limits: false
apply_to_motor: true"`

预期：
- 服务 `success=false`。
- `message` 提示当前位置不在请求限位范围内。
- 不应下发新的限位设置帧。

---

### T6 启动阶段位置匹配（上电保护）

步骤：
1. 配置 `startup_position_sync_timeout_sec`（例如 `1.0`）。
2. 构造“当前实际位置超出配置限位”的场景。
3. 重启 `can_driver_node`。

预期：
- 节点启动报错并拒绝激活（fail-safe）。
- 日志包含 startup position out of limits。

---

### T7 非位置关节拒绝设置

步骤：
1. 对 `velocity` 关节调用 `set_zero_limit`。

预期：
- `success=false`。
- `message` 明确提示仅支持位置关节。

---

### T8 非PP协议兼容行为

步骤：
1. 对 MT 关节调用并设置 `apply_to_motor=true`。

预期：
- 若协议不支持物理下发，服务返回失败且提示“协议不支持/拒绝”。
- 系统无崩溃、无死锁。

## 4. 通过标准

- T1~T8 全部满足预期。
- 关键安全项（T5/T6）必须通过。
- 无异常退出、无持续错误刷屏。

## 5. 失败排查建议

1. 若 `set_zero_limit` 不存在：确认已重新编译并 source 工作空间。
2. 若 `use_urdf_limits=true` 失败：确认 `robot_description` 存在、关节名一致。
3. 若抓不到寄存器设置帧：确认目标关节是 PP 协议，且 `apply_to_motor=true`。
4. 若重复失败：先执行 `recover`，再重试设置。
