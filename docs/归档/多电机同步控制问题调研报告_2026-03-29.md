# 多电机同步控制问题调研报告

**日期**: 2026-03-29
**问题描述**: PP 模式下多电机起步和结束时间不一致，时间差约 100ms~300ms
**调研目标**: 定位问题根因，评估协议支持能力，提出改进方向

---

## 一、问题现象

在使用 Profile Position (PP) 模式同时驱动多个电机时，观察到以下现象：

- **起步不同步**: 多个电机开始运动的时间差在 100ms~300ms 之间
- **结束不同步**: 由于起步时间不一致，结束时间也相应错开
- **预期行为**: 协议层应支持同步模式，多轴应能同时触发运动

---

## 二、问题根因分析

### 2.1 核心问题：缺少协议级同步触发机制

当前 `can_driver` 的实现是**逐轴串行下发命令**，没有"预装载 + 统一触发"的机制。

**关键代码位置**: [CanDriverHW.cpp:693](../src/CanDriverHW.cpp#L693)

```cpp
// write() 方法按组、按电机串行调用
for (const auto& group : jointGroups_) {
    for (size_t i = 0; i < group.jointIndices.size(); ++i) {
        // 逐个电机调用 setPosition()
        proto->setPosition(motorId, targetPos, targetVel);
    }
}
```

**问题**: 每个电机的命令是立即生效的，不是"装填后等待统一触发"。

### 2.2 PP 协议层的额外帧开销

**关键代码位置**: [EyouCan.cpp:200](../src/EyouCan.cpp#L200)

PP 的 `setPosition()` 每次调用会发送多个 CAN 帧：
- 普通模式：发送 2 帧（速度 0x09 + 位置 0x0A）
- Fast Write 模式：发送 4 帧

**时间累积效应**:
- 4 轴电机 × 2 帧/轴 = 8 帧/周期（普通模式）
- 4 轴电机 × 4 帧/轴 = 16 帧/周期（Fast Write）
- 假设 CAN 总线 500kbps，每帧约 0.2ms，8 帧就是 1.6ms 的串行延迟

**问题**: 第 1 个电机和第 4 个电机的命令下发时刻天然就有 1.6ms 以上的差异。

### 2.3 协议语义不稳定

**代码与测试/文档不一致**:

- **源码实现** [EyouCan.cpp:200](../src/EyouCan.cpp#L200): 位置命令会先发速度帧 0x09，再发位置帧 0x0A
- **测试预期** [test_eyou_can_protocol.cpp:58](../tests/test_eyou_can_protocol.cpp#L58): 位置命令只应发 0x0A
- **文档说明** [使用指南.md:45](使用指南.md#L45): 也倾向于"位置命令只写 0x0A"

**影响**: PP 链路的命令语义还在演进中，时序问题不只是"总线慢"，还有"命令定义不稳定"。

### 2.4 后台轮询线程持续占用总线

**关键代码位置**: [EyouCan.cpp:636](../src/EyouCan.cpp#L636)

`refreshMotorStates()` 每个电机会轮询 6 个状态：
- 位置、模式、使能、故障、电流、速度

**配置参数**: [can_driver.yaml:23](../config/can_driver.yaml#L23)
```yaml
motor_query_hz: 100.0  # 每秒 100 次轮询
```

**总线占用估算**:
- 4 轴 × 6 个读请求/轴 × 100Hz = 2400 帧/秒
- 平均每 0.4ms 就有一个轮询帧

**问题**: 前台写命令和后台轮询在时间上互相穿插，进一步拉长了多轴命令的实际下发时间窗口。

### 2.5 发送侧缺少批量互斥

**关键代码位置**: [SocketCanController.cpp:87](../src/SocketCanController.cpp#L87)

`send()` 方法只是单帧写 socket，没有"批量命令原子性"保护。

**现状**:
- `CanDriverHW::write()` 有 device mutex，但只包在调用 proto 的外层
- PP 协议自己的刷新线程不走这把锁
- 结果：写命令和轮询读可能在 CAN 帧层面交错发送

**影响**: 本来就串行的命令流，被轮询帧再次切碎，雪上加霜。

## 三、协议支持能力评估

### 3.1 意优协议本身支持同步模式

**协议文档**: [意优行星关节CAN通讯协议.md](意优行星关节CAN通讯协议.md)

协议明确定义了同步相关能力：

1. **广播地址**: `CAN_ID = 0x00` 是广播地址
2. **同步命令**: `CMD = 0x00` 用于"心跳、网络、同步"
3. **同步模式**: `0x0F` 工作模式中，值 `5` 是"周期同步位置模式"（CSP）

**结论**: 协议层面完全支持你提到的"PP 应该支持同步模式"，问题在于 `can_driver` 没有把这个能力实现出来。

### 3.2 当前 can_driver 未实现 CSP 模式

**抽象层**: [CanProtocol.h:15](../include/can_driver/CanProtocol.h#L15)
```cpp
enum class ControlMode {
    Position = 0,  // PP 模式
    Velocity = 1   // PV 模式
    // 缺少 CSP 模式定义
};
```

**PP 实现**: [EyouCan.cpp:151](../src/EyouCan.cpp#L151)
- 只把模式写成 `1`（PP）或 `3`（PV）
- 没有模式 `5`（CSP）的处理逻辑

**服务入口**: [MotorCommand.srv](../srv/MotorCommand.srv) 和 [CanDriverHW.cpp:1087](../src/CanDriverHW.cpp#L1087)
- 只接受 `0`（Position）或 `1`（Velocity）
- 没有 CSP 的服务接口

**结论**: "PP 本身能用 CSP"这件事，在现代码里还只是协议文档层面的能力，驱动层完全没有打通。

### 3.3 Eyou_Canopen_Master 的架构更接近正确方向

**关键架构组件**:

4. **CIA402 状态机** [cia402_state_machine.cpp:300](../../Eyou_Canopen_Master/src/cia402_state_machine.cpp#L300)
   - 已经把 CSP/CSV/CST 当一等公民

**结论**: `Eyou_Canopen_Master` 的架构已经把"同步周期、命令门控、模式切换、安全 target"这些关键件都搭好了，可以作为改进 `can_driver` 的参考。

## 四、问题成因总结

### 4.1 起步不一致的主因

1. **串行下发**: `can_driver` 逐轴串行下发命令，没有协议级同步触发
2. **额外帧开销**: PP 每轴位置命令发送 2~4 帧，累积延迟明显
3. **总线占用**: 后台轮询线程持续插流量，拉长命令下发窗口
4. **缺少批量互斥**: 写命令和轮询读在 CAN 帧层面交错

### 4.2 结束不一致的主因

各轴实际开始时刻不同，PP 内部又是各自本地轨迹发生器，所以起跑错开后，收尾也会跟着错开。

### 4.3 根本原因

**这不是单纯把 ros_control 频率再调高能解决的问题**，因为问题点在协议层触发语义，而不是上层控制周期。

## 五、改进方向建议

### 5.1 短期方向：补充协议级同步触发

**目标**: 让 PP 模式支持"预装载 + 广播同步"

**需要补充的能力**:
1. 在 `CanProtocol` 抽象层增加"装填命令"和"触发执行"两个独立接口
2. 在 `EyouCan` 实现层支持：
   - 装填：发送位置目标但不立即执行
   - 触发：发送广播同步帧（`CAN_ID=0x00, CMD=0x00`）
3. 在 `CanDriverHW::write()` 改为：
   - 先遍历所有电机装填命令
   - 最后发送一次广播同步触发

**预期效果**: 多轴起步时间差从 100~300ms 降低到 < 5ms（取决于 CAN 总线延迟）

### 5.2 中期方向：支持 CSP 模式（推荐方案）

**目标**: 把协议文档里的 CSP 能力在驱动层打通

**CSP 模式的实际工作原理**（基于意优官方 SDK 示例代码）:

CSP 模式不是"预装载 + 广播同步"，而是：
1. 上层控制器以固定高频周期（如 4ms/250Hz）持续发送位置目标
2. 使用 **CMD=0x05 快写命令**（`quick_setTargetPosition`），立即生效，无需返回确认
3. 通过高频位置更新实现平滑的轨迹跟踪

**官方示例代码流程**:
```cpp
// 1. 设置 CSP 模式
planet_setMode(devIndex, id, 5);

// 2. 使能电机
planet_setEnabled(devIndex, id, true);

// 3. 设置目标速度（CSP 模式下仍需设置）
planet_setTargetVelocity(devIndex, id, 5);

// 4. 循环发送位置命令（4ms 周期）
while (1) {
    planet_quick_setTargetPosition(devIndex, id, pos);
    std::this_thread::sleep_for(std::chrono::milliseconds(4));
    pos += 0.1;  // 位置增量
}
```

**多轴同步的实现方式**:
```cpp
while (1) {
    // 在同一个控制周期内，快速给所有电机发送位置
    quick_setTargetPosition(id1, pos1);
    quick_setTargetPosition(id2, pos2);
    quick_setTargetPosition(id3, pos3);
    quick_setTargetPosition(id4, pos4);

    sleep(4ms);  // 控制周期

    // 更新下一周期的位置目标
    pos1 += delta1;
    pos2 += delta2;
    // ...
}
```

**需要补充的接口**:
1. `ControlMode` 枚举增加 `CSP = 5`
2. `CanProtocol` 增加 `quickSetPosition()` 接口（对应 CMD=0x05 快写命令）
3. `EyouCan::quickSetPosition()` 实现：
   - 使用 CMD=0x05（快写命令）
   - 只发送位置帧，无需等待返回
4. `CanDriverHW::write()` 在 CSP 模式下：
   - 快速遍历所有电机调用 `quickSetPosition()`
   - 不需要广播同步帧
5. 上层控制器需要以固定高频周期（建议 4ms）调用 `write()`

**CSP 的优势**:
- 不依赖电机内部轨迹发生器，上层控制器有完全的轨迹控制权
- 使用快写命令（CMD=0x05），无需等待返回，总线效率高
- 通过高频更新（250Hz）实现平滑控制
- 多轴起步时间差取决于串行发送延迟，预计 < 10ms（4 轴 × 1 帧/轴 × 0.2ms/帧 = 0.8ms）

**与 PP 模式的对比**:
| 特性 | PP 模式（模式 1） | CSP 模式（模式 5） |
|------|------------------|-------------------|
| 轨迹生成 | 电机内部 | 上层控制器 |
| 命令类型 | CMD=0x01（写入命令，需返回） | CMD=0x05（快写命令，无需返回） |
| 每轴帧数 | 2~4 帧 | 1 帧 |
| 控制周期 | 不固定 | 固定高频（如 4ms） |
| 同步精度 | 100~300ms | < 10ms |

### 5.3 架构方向：借鉴 Eyou_Canopen_Master

**建议参考的设计模式**:

1. **命令快照机制**
   - 把"命令下发"和"状态读取"解耦
   - 用 SharedState 做线程间数据面

2. **同步序列管理**
   - 统一的周期调度器
   - 确保命令的时间一致性

3. **Guard 机制**
   - 命令有效性检查
   - 防止过时命令被执行

4. **状态机驱动的模式切换**
   - 用 CIA402 状态机管理电机生命周期
   - 安全的模式切换流程

**实施建议**: 不要一次性重构整个架构，可以先在 PP 协议层补同步能力，验证效果后再逐步引入更完整的架构模式。

```cpp
bool EyouCan::quickSetPosition(uint8_t motorId, double position) {
    // 使用 CMD=0x05 快写命令
    // 只发送位置帧，无需等待返回
    // 位置转换：角度 -> 脉冲数（65536 脉冲 = 360°）
    int32_t pulses = static_cast<int32_t>(position / 360.0 * 65536);

    CanFrame frame;
    frame.can_id = motorId;
    frame.can_dlc = 8;
    frame.data[0] = 0x05;  // CMD: 快写命令
    frame.data[1] = 0x0A;  // ADDR: 目标位置寄存器
    // data[2-5]: 位置值（大端模式）
    frame.data[2] = (pulses >> 24) & 0xFF;
    frame.data[3] = (pulses >> 16) & 0xFF;
    frame.data[4] = (pulses >> 8) & 0xFF;
    frame.data[5] = pulses & 0xFF;
    frame.data[6] = 0x00;  // NC
    frame.data[7] = 0x00;  // NC

    return transport_->send(frame);
}
```

**风险 1**: 快写命令的 CAN 帧格式可能与预期不符

**应对**:
- 先用官方 SDK 写测试程序，抓包验证帧格式
- 参考官方示例代码的实现逻辑

**风险 2**: CSP 模式下电机行为可能与文档不一致

**应对**:
- 先做单轴测试，确认电机响应
- 逐步增加轴数，观察同步效果

**风险 3**: 高频控制（250Hz）可能增加 CPU 负载

**应对**:
- 监控 CPU 使用率
- 如有必要，调整控制频率到 100Hz（10ms 周期）
