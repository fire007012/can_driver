# CSP 模式稳定性评估报告

日期：2026-03-31

## 1. 评估目的

本文档回答一个核心问题：

- `can_driver` 当前这套实现，是否已经达到“CSP 模式下可稳定运行”的程度？

这里的“稳定运行”不是指“偶尔能跑起来”，而是指同时满足以下几点：

- 在 `control_frequency=100Hz` 一类常见控制频率下可持续下发位置命令
- 从站在线时不会因为协议层自身实现问题长期误判状态
- 生命周期主链、轴级闭环、安全策略三者之间语义一致
- 调试接口与状态发布能真实反映当前模式和健康状态
- 多轴运行时不会因为发送策略或状态刷新策略出现明显结构性缺陷

## 2. 结论摘要

结论先行：

- 当前版本已经具备 `CSP` 的基础数据面能力，可以作为“联调版 / 验证版”继续推进。
- 但它还不能被评估为“CSP 模式下稳定运作”。
- 当前状态更接近“黄灯”而不是“绿灯”。

更具体地说：

- 能力上：已经支持 `control_mode: csp`、`setMode(CSP)`、`quickSetPosition()`、非阻塞发送、每总线单 worker 的发送调度、基础烟测覆盖。
- 风险上：协议状态回读、共享状态闭环、生命周期健康判据、模式可观测性这几块还存在核心缺口。
- 结论上：短时联调和小范围实验可以继续做，但不建议把它视为已经成熟的稳定 `CSP` 方案。

## 3. 当前已经具备的能力

### 3.1 数据面基础已经成形

当前 `can_driver` 在数据面上已经具备几项重要基础：

- `PP` 协议支持 `CSP` 模式枚举和值
- `EyouCan::quickSetPosition()` 已实现，`CSP` 命令走快写
- `CanDriverHW` 已支持 `control_mode: csp`
- `yaml` 中可直接填写 `position_scale: 65536` 这类 PPR 语义
- 发送链路已改为非阻塞
- `DeviceRuntime` 已统一管理每个 CAN 设备的发送队列
- 控制帧队列采用“丢最旧、保最新”的策略，方向正确

这些能力意味着：

- 包已经不再停留在“完全不能做 CSP”的阶段
- 主链上最关键的几段代码已经连起来了
- 后续问题主要集中在“状态闭环和运行策略”，而不再是“有没有接口”

### 3.2 当前负载量级本身并不离谱

按当前默认配置粗算：

- `control_frequency = 100Hz`
- `motor_query_hz = 20Hz`
- 默认启用 3 个 `PP + CSP` 关节

则大致为：

- 控制写帧：约 `3 * 100 = 300` 帧/秒
- 查询帧：位置、速度每周期查询；模式、使能、故障、电流低频查询
- 平均查询量级：约 `120 ~ 150` 帧/秒

总量大致在 `400+` 帧/秒这个级别。

这个量级说明一个重要事实：

- 当前最核心的问题不是“总线带宽一定不够”
- 而是“状态刷新和闭环判据还不够可靠”

也就是说，它的主要瓶颈更像是架构和时序问题，而不是纯粹的流量问题。

## 4. 当前不能判定为稳定的核心原因

### 4.1 `CSP` 模式回读语义存在关键缺口

当前协议层虽然可以下发 `setMode(CSP)`，但模式回读路径还没有把驱动返回的 `0x05` 明确识别为 `CSP`。

这会带来两个后果：

- 本地协议状态缓存可能把 `CSP` 误当成普通 `Position`
- `SharedDriverState -> AxisRuntime -> LifecycleDriverOps` 这条闭环链路可能因此长期认为“模式未就绪”

这不是显示层小问题，而是会直接影响：

- `motionHealthy()` 判定
- `RequestRelease()` 是否成功
- 运行中模式匹配是否被认为成立

换句话说：

- 命令面已经能发 `CSP`
- 但状态面还没有完整理解 `CSP`

这是当前最核心的稳定性缺口之一。

### 4.2 状态查询频率与控制需求不匹配

当前 `EyouCan` 的刷新策略是：

- 位置、速度：每个 refresh 周期都查
- 模式、使能、故障、电流：每 `kSlowDivider` 个周期查一次

而 `kSlowDivider` 目前是 `50`。

在 `motor_query_hz = 20Hz` 时，意味着：

- `mode / enable / fault / current` 的刷新周期约为 `2.5s`

这对 `CSP` 是明显过慢的，原因有三点：

- 生命周期切换依赖这些状态是否新鲜
- 安全策略需要尽快看到 `enabled` / `fault` 的真实变化
- 模式切换后如果 `mode` 回读慢，系统会长时间处于“状态不确定”

因此，即便总线流量本身不算高，当前刷新策略仍会让 `CSP` 运行态带有明显的“陈旧状态”问题。

### 4.3 当前实现更像“主机定时快写位置流”，而不是强同步意义上的 CSP

从架构上看，当前实现是：

- 主线程 `read -> update -> write`
- `write()` 按关节逐个准备 raw command
- 对 `csp` 关节逐个调用 `quickSetPosition()`
- 命令进入 `DeviceRuntime` 队列
- 单个 worker 再串行发出去

这套模型是可工作的，但它更接近：

- “高频位置流式控制”

而不是严格意义上的：

- “多轴同一时基下的强同步 `CSP`”

它和 `lely` 风格实现的主要差异在于：

- 没有统一同步边沿
- 没有总线级统一周期调度
- 没有多个轴在同一拍上被协议层原生组织成强同步动作

所以结论应该说清楚：

- 它可以朝“工程上可用的 `CSP`”继续收敛
- 但现在还达不到 `lely` 式同步确定性

### 4.4 对外状态可观测性不足

当前 `motor_states` 话题里的 `mode` 只有：

- `MODE_POSITION`
- `MODE_VELOCITY`

没有独立的：

- `MODE_CSP`

这会导致几个问题：

- UI 看不出轴当前到底是 `position` 还是 `csp`
- 日志与 topic 状态不一致时，排障成本高
- 后续如果要比较 `PP` 和 `CSP` 行为，缺少可观测字段

这不是“能不能动”的问题，但它会显著影响“能不能稳定维护和排障”。

### 4.5 现有测试还没有真正打通完整的 `CSP` 闭环主链

当前测试状态是：

- `CSP` 协议编码测试已经有了
- `CanDriverHW` 的 `CSP + PPR` 数据面 smoke test 已经有了
- `AxisRuntime` 和 `LifecycleDriverOps` 的闭环测试也已经有了

但仍有一个关键空缺：

- 还没有一条真正的端到端测试把
  `CanDriverHW -> DeviceManager -> SharedDriverState -> AxisRuntime -> RequestEnable/RequestRelease`
  的 `CSP` 路径一次性打通

这意味着：

- 当前测试已经足够发现不少局部回归
- 但还不足以证明“整个 `CSP` 闭环已经稳定”

## 5. 当前能力评估

### 5.1 可以做到什么

当前版本可以 reasonably 期待做到：

- `yaml` 中配置 `control_mode: csp`
- 启动时自动设置 `CSP` 模式
- 按控制周期持续下发快写位置命令
- 从站掉线或链路异常时不再阻塞整个发送线程
- 在小规模联调中观察轨迹跟随和基础动作

因此如果目标是：

- 单轴 / 小规模多轴联调
- 验证 `CSP` 命令链是否通
- 排查机械侧或协议侧基础问题

那么当前版本已经值得继续用。

### 5.2 还做不到什么

当前版本暂时不能放心承诺：

- `CSP` 生命周期闭环已经完全可信
- 模式匹配判据已经正确覆盖 `CSP`
- 安全策略能基于足够新鲜的模式 / 使能 / 故障状态工作
- 多轴同步行为已接近 `lely`
- 外部 topic / UI 能准确反映 `CSP` 运行状态

所以如果目标是：

- 长时间稳定运行
- 高可靠生产使用
- 接近工业 `CSP` 的同步一致性

那么当前版本还不能算达标。

## 6. 评估结论分级

建议使用下面这个分级来描述当前状态：

- 绿色：可以稳定长期运行，闭环判据完整，状态可观测，测试充分
- 黄色：基础链路可用，但存在会影响稳定性的结构性缺口
- 红色：关键链路未打通，无法进行有效联调

当前 `can_driver` 的 `CSP` 状态应评估为：

- 黄色，偏深

理由：

- 不是红色，因为数据面和发送面已经能工作
- 不是绿色，因为模式回读、状态刷新节拍、闭环一致性、端到端验证仍有关键缺口

## 7. 建议的下一步

建议按优先级分成两层推进。

### 7.1 第一优先级：先把“能不能稳定跑”补齐

这几项做完之前，不建议宣称 `CSP` 已稳定：

1. 修复 `0x0F` 模式回读对 `CSP` 的识别
2. 补 `CSP` 模式回读与 `modeMatched` 的测试
3. 提高 `mode / enabled / fault` 的刷新频率
4. 增加 `CanDriverHW + SharedDriverState + Lifecycle` 的端到端 `CSP` 测试
5. 给 `MotorState.msg` 增加 `MODE_CSP`

### 7.2 第二优先级：再决定是否追 `lely` 级同步

如果目标只是：

- “机械臂多轴能稳定流式运行”

那在第一优先级修完后，很可能已经够用。

如果目标是：

- “尽量接近 `lely` 的同步确定性和运行质感”

那还需要进一步做：

- per-device 周期调度收口
- 更明确的多轴同拍发送策略
- 进一步降低查询流量对控制流的干扰
- 重新设计更接近同步总线时序的运行时

## 8. 最终判断

最终判断如下：

- 当前 `can_driver` 已经具备 `CSP` 的基础能力
- 但还不能被认定为“`CSP` 模式下稳定运作”
- 当前最需要修的不是“再加更多功能”，而是“把状态闭环和刷新策略做正确”

一句话概括：

- 现在这套实现已经能继续推进 `CSP` 联调
- 但距离“稳定、可信、可长期运行的 `CSP`”还差一段，而且差的正是核心闭环部分
