# can_driver 生命周期主链重构计划

## Summary

本文档用于落实 `can_driver` 侧生命周期主链重构，目标是让其对外服务语义与
`Eyou_Canopen_Master` 保持一致，并与项目级文档
`/home/dianhua/Robot24_catkin_ws/src/docs/统一外观层实施方案.md`
中的约束对齐。

本轮重构只聚焦 `can_driver` 包内，不同步实现统一外观层节点；重点是把
`CanDriverHW` 从“生命周期服务入口 + 状态机 + 安全闸门 + RobotHW 桥接”的混合角色，
收口为清晰分层：

- `LifecycleServiceGateway`：只负责 ROS service 入口与兼容别名
- `OperationalCoordinator`：负责状态迁移、动作编排、故障降级
- `CommandGate`：负责 hold、fresh-command、掉线恢复后的命令同步
- `CanDriverHW`：只负责 `RobotHW` 读写桥接、direct topic、状态发布、维护类服务

## 目标与约束

本次重构完成后，`can_driver` 需要满足以下约束：

- `recover` 不再等价于 `Enable`
- `Running` 之外不得下发正常运动命令
- `recover`、`enable`、`resume`、设备掉线恢复后都必须要求 fresh command
- 生命周期服务语义与 `Eyou_Canopen_Master` 一致
- 旧接口只保留兼容入口，不再保留旧语义

统一生命周期状态如下：

- `Inactive`
- `Configured`
- `Standby`
- `Armed`
- `Running`
- `Faulted`
- `Recovering`
- `ShuttingDown`

状态语义固定为：

- `init`：建立通信并完成启动检查，成功后进入 `Armed`
- `enable`：`Standby -> Armed`，只表示进入可运行待命态，不直接开始运动
- `resume`：`Armed -> Running`
- `halt`：`Running -> Armed`
- `disable`：`Standby / Armed / Running -> Standby`
- `recover`：仅 `Faulted -> Standby`，不得隐式 `enable` 或 `resume`
- `shutdown`：停止通信并回到 `Configured`

## 分层设计

### 1. LifecycleServiceGateway

新增 `LifecycleServiceGateway`，从 `CanDriverHW` 中移出以下生命周期服务：

- `init`
- `shutdown`
- `recover`
- `enable`
- `disable`
- `halt`
- `resume`

该层只负责：

- 注册 ROS service
- 参数合法性检查
- 兼容旧服务别名
- 调用 `OperationalCoordinator`
- 将结果映射到 ROS response

该层不再持有业务状态，也不直接访问协议层。

### 2. OperationalCoordinator

将 `OperationalCoordinator` 从“轻量状态寄存器”提升为真正的生命周期协调器。

通过注入 `DriverOps` 回调集合，让 coordinator 负责：

- 合法状态迁移判断
- 生命周期动作编排
- 动作失败时的原子回滚
- `Armed/Running` 下的自动故障降级
- 恢复后的命令重同步触发

`DriverOps` 至少需要包含以下能力：

- `init_device`
- `enable_all`
- `disable_all`
- `halt_all`
- `reset_all_faults`
- `shutdown_all`
- `all_motion_healthy`
- `all_faults_cleared`
- `request_hold_and_fresh_command`

关键行为要求：

- `RequestEnable()` 若部分电机 enable 成功、部分失败，必须回滚已成功电机到 disabled，
  生命周期保持在 `Standby`
- `RequestDisable()` 与 `RequestHalt()` 必须返回真实执行结果，不允许“动作失败但 service 成功”
- `RequestRecover()` 仅允许全局恢复，失败时保持 `Faulted`
- `UpdateFromFeedback()` 统一负责自动从 `Armed/Running` 降级到 `Faulted`

### 3. CommandGate

新增 `CommandGate`，统一管理所有“进入安全保持态”的命令逻辑，避免继续在
`CanDriverHW` 内部复制状态与缓冲处理。

职责包括：

- 生命周期切换后的 hold
- fresh-command latch
- device not ready 时清命令和阻断输出
- 设备恢复后的一拍 hold
- 命令 baseline 管理

以下路径必须统一进入 fresh-command 保护：

- `init`
- `enable`
- `resume`
- `recover`
- 设备掉线恢复

### 4. CanDriverHW

重构后的 `CanDriverHW` 只保留：

- 参数读取与 joint 配置初始化
- `read()` 中的反馈同步
- `write()` 中的命令采样、限幅、缩放与协议写入
- direct topic 订阅
- `motor_states` 发布
- `motor_command`
- `set_zero_limit`

明确移出的内容：

- 生命周期 service 注册
- 生命周期状态迁移
- `recover/enable/disable/halt/resume/shutdown` 编排逻辑
- fresh-command latch 的内部状态管理
- device recover 策略决策

`publishMotorStates()` 不再直接修改 lifecycle mode，而是把健康信息交给
`OperationalCoordinator` 统一处理。

## 接口与兼容策略

### 生命周期服务

保留以下对外生命周期服务名：

- `init`
- `shutdown`
- `recover`
- `enable`
- `disable`
- `halt`
- `resume`

接口形状策略：

- `enable/disable/halt/resume` 继续使用 `std_srvs/Trigger`
- `init` 和 `shutdown` 保持现有 srv 形状，因为它们带参数
- `recover` 本轮先保留 `can_driver/Recover.srv`，避免立即引入 breaking change

### recover 兼容规则

`recover` 的公开语义从本轮开始固定为“全局生命周期恢复”，不再承载 per-motor 语义。

兼容规则如下：

- `motor_id == 0xFFFF`：视为合法全局恢复
- 指定具体 `motor_id`：直接返回失败，并明确提示
  “per-motor recover 已废弃，请使用全局 recover”

说明：

- 这是兼容旧 wire API，但不兼容旧业务语义
- 若后续仍需要单电机 fault reset，只能通过维护类接口单独提供，
  不能继续复用 lifecycle `recover`

### 调试与维护接口

以下接口本轮保留：

- `motor_command`
- `set_zero_limit`
- per-joint direct topic

但要明确：

- 它们属于维护/调试或底层兼容入口
- 不承担统一生命周期主语义
- 文档中不得再把它们描述为正式生命周期控制入口

## Commit 划分

### Commit 1

`refactor(can_driver): split lifecycle services from CanDriverHW`

目标：

- 引入 `LifecycleServiceGateway`
- 把生命周期 service 注册和 handler 从 `CanDriverHW` 中移出
- 保持外部服务名与基本行为不变

完成标准：

- `CanDriverHW` 不再声明和实现生命周期 service handler
- gateway 独立注册 `init/enable/disable/halt/resume/recover/shutdown`

### Commit 2

`refactor(can_driver): promote operational coordinator to own lifecycle actions`

目标：

- 为 `OperationalCoordinator` 注入 `DriverOps`
- 将生命周期动作编排迁入 coordinator
- 修复当前语义漏洞

必须修复的问题：

- `enable` 部分成功后失败，必须回滚
- `disable` / `halt` 不能假成功
- `recover` 只能做 `Faulted -> Standby`
- `recover` 不得再隐式 enable

### Commit 3

`refactor(can_driver): extract command gate and unify hold semantics`

目标：

- 抽出 `CommandGate`
- 收敛重复的 hold 和命令清理逻辑
- 简化 `CanDriverHW::write()`

必须统一的路径：

- lifecycle transition hold
- startup command alignment
- device not ready
- device recover hold
- fresh-command latch

### Commit 4

`refactor(can_driver): route health updates through coordinator`

目标：

- 将 fault/device health 的 lifecycle 降级入口收敛到 coordinator
- `CanDriverHW` 不再直接切换 lifecycle mode

完成标准：

- `publishMotorStates()` 只上报健康快照
- `write()` 只消费 coordinator/gate 的结果，不再自己维护第二套状态机

### Commit 5

`test(can_driver): add lifecycle and command gate coverage`

目标：

- 为新分层补齐测试
- 统一文档与脚本中的恢复语义

完成标准：

- 新增 coordinator 单测
- 扩展 `CanDriverHW` smoke test
- README、使用指南、脚本统一切到全局 `recover`

## 测试计划

新增 `test_operational_coordinator.cpp`，至少覆盖：

- 合法/非法状态迁移矩阵
- `Enable` 部分失败时已启用电机被回滚
- `Disable` / `Halt` 任一动作失败时返回失败
- `Recover` 成功时 `Faulted -> Standby`
- `Recover` 失败或 fault 未清时保持 `Faulted`
- `Armed/Running` 下自动故障降级到 `Faulted`

扩展 `test_can_driver_hw_smoke.cpp`，至少覆盖：

- `Running` 之外 `write()` 不下发正常运动命令
- `halt -> resume` 后旧命令不会直接透传
- `recover` 后未收到 fresh command 时运动保持阻断
- 设备掉线恢复后必须收到 fresh command 才重新放行
- `recover` 传具体 `motor_id` 时返回 legacy rejected

构建与验收命令：

- `catkin_make --pkg can_driver`
- `catkin_make run_tests_can_driver`

## Assumptions

- 本轮只做 `can_driver` 主链重构，不同步实现统一外观层节点
- `motor_command` 与 `set_zero_limit` 暂不拆成独立 gateway，但要复用统一的底层执行器
- `Recover.srv` 本轮不做 breaking change；真正改成 `Trigger` 或新 srv 形状，留到统一外观层对外 API 收敛阶段
- 项目级总方案仍以
  `/home/dianhua/Robot24_catkin_ws/src/docs/统一外观层实施方案.md`
  为准，本文件是 `can_driver` 包内的专项落地计划
