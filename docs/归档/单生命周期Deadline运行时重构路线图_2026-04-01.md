## 1. 目标

本路线图用于收敛 `can_driver` 当前架构，使其满足以下目标：

- 对外生命周期语义继续与 `Eyou_Canopen_Master` 保持一致
- 内部只保留一个系统级生命周期状态机：`OperationalCoordinator`
- PP 协议反馈轮询从“按轮次轮转”改为“按字段 deadline 调度”
- 每个关键反馈字段都具备明确的最大查询间隔约束
- 告警从“回包年龄”改为“错过应有刷新窗口”
- 不再让私有协议驱动继续向“第二套仿 CiA402 轴状态机”演化

## 2. 当前问题归纳

### 2.1 生命周期主链已基本对齐，但运行时未彻底对齐

当前 `can_driver` 已具备：

- `LifecycleServiceGateway`
- `OperationalCoordinator`
- `LifecycleDriverOps`
- `SharedDriverState`
- `CommandGate`

这意味着：

- 顶层系统生命周期入口已经存在
- 顶层系统状态集合已经基本与 `Eyou_Canopen_Master` 对齐

但真正不稳定的部分在于：

- 刷新调度仍然是按 `cycle % N` 的轮盘式计划
- 协议层的发送、超时、回包 freshness、告警语义仍然分散在多层
- 轴级 readiness 判据已经有向“第二套轴状态机”扩张的趋势

### 2.2 PP 刷新调度天然无法保证字段 freshness

当前稳态 PP 刷新计划不是“每字段周期调度”，而是“每轮选择部分字段查询”。

结果包括：

- `Position` 与 `Velocity` 的刷新频率明显高于其余字段
- `Mode / Enable / Fault / Current` 会被轮转降频
- 压力模式下某些字段甚至可能长时间得不到调度

这会直接破坏以下上层语义：

- `feedback_fresh`
- `mode_ready`
- `enabled_ready`
- `fault_cleared`
- `axis_ready_for_run`

### 2.3 stale warning 语义与调度语义不一致

当前 stale warning 会把：

- “很久没收到某字段新回包”

误当作：

- “该字段异常超时”

但实际上很多字段只是：

- 本轮根本没被调度
- 被压力模式降载
- 被 query in-flight / backoff 机制延后

因此 warning 需要重写成：

- “该字段已经到达查询 deadline，并连续错过若干刷新窗口”

而不是：

- “距离上次回包已经过去很久”

## 3. 目标架构

建议将 `can_driver` 收敛到以下主链：

`LifecycleServiceGateway -> OperationalCoordinator -> LifecycleDriverOps -> DeviceManager -> DeviceRuntime -> EyouCan/MtCan -> SharedDriverState`

职责边界固定如下：

- `OperationalCoordinator`
  - 只负责系统级生命周期状态迁移与动作编排
- `LifecycleDriverOps`
  - 只负责把系统动作映射到协议动作，并聚合轴级准入判据
- `DeviceManager`
  - 只负责 device 级 transport / protocol / refresh runtime 组织
- `DeviceRuntime`
  - 只负责一个 device 的统一发包所有权、优先级与背压处理
- `EyouCan / MtCan`
  - 只负责协议编解码、读写请求登记、回包解析、事实回填
- `SharedDriverState`
  - 只保存共享事实与派生观测，不保存第二套生命周期状态
- `AxisReadinessEvaluator`
  - 只输出准入判据，不输出第二套轴级状态名

## 4. 关键原则

### 4.1 唯一生命周期原则

系统内唯一具备生命周期状态名解释权的对象只有：

- `OperationalCoordinator`

轴级层不再拥有：

- `Armed`
- `Running`
- `Faulted`
- `Recovering`

这类命名状态。

### 4.2 字段 deadline 原则

PP 协议关键反馈字段的调度应从“按轮次”改为“按字段最大间隔”。

第一阶段默认目标：

- `Position`: `50ms`
- `Velocity`: `100ms`
- `Mode`: `100ms`
- `Enable`: `100ms`
- `Fault`: `100ms`
- `Current`: `100ms`

含义是：

- 任何字段都不能因为轮盘调度而长期饥饿
- 压力模式只能降低同一时刻的发包预算，不能取消字段的 deadline 责任

### 4.3 告警语义原则

后续告警只允许表达以下事实：

- 某字段已经到达或超出 deadline
- 调度器已经尝试安排查询
- 但连续若干刷新窗口仍未收到新包

不再允许直接把：

- `now - lastResponse`

解释为“设备超时”。

### 4.4 SharedState 收缩原则

`SharedDriverState` 只保存：

- 轴反馈事实
- 轴命令事实
- 轴 intent
- 设备健康
- 派生观测时间戳

后续 deadline 调度器内部状态，例如：

- `next_due`
- `last_due`
- `last_issue`
- `missed_windows`
- `backoff`

不进入 `SharedDriverState`。

## 5. 分阶段实施

### Phase 1：先把运行时稳定性做正确

目标：

- PP 刷新真正满足 freshness 约束
- warning 不再误报
- 生命周期上层不再被虚假的 freshness 污染

#### Commit A1：文档冻结

产出：

- 本路线图文档

验收：

- 统一目标架构、边界、字段 deadline 与 warning 语义

#### Commit A2：PP deadline 调度器

改动目标：

- 替换当前 `RefreshScheduler` 对 PP 的 `cycle % N` 轮转逻辑
- 引入每轴每字段的 deadline 调度状态
- 调度器按“到期优先”生成查询请求

边界：

- 先只改 PP
- 暂不改 MT
- 暂不改变外部 service / UI / YAML 语义

验收：

- 单轴、多轴情况下，PP 六个字段的调度间隔均有明确上限

#### Commit A3：调度 / 发送 / 回包三阶段分离

改动目标：

- 显式区分：
  - 该字段应该被查询
  - 该字段本轮实际发送成功
  - 该字段收到新回包

验收：

- 任何 freshness 判定都不再偷用单一 `lastResponse`

#### Commit A4：基于 missed windows 的 warning

改动目标：

- 用“连续错过刷新窗口”替代“回包年龄告警”

验收：

- 不再出现 `8036ms`、`10000ms` 这种对低频字段误导性的 stale 告警

#### Commit A5：压力模式改为限预算，不再饿死字段

改动目标：

- 保留 pressure mode
- 但 pressure mode 只削减单周期发包预算，不取消字段 deadline

验收：

- 在背压场景下，字段刷新间隔仍然受控，不出现长时间完全不查

#### Commit A6：测试封口

补充测试：

- deadline 调度单测
- 多轴场景 freshness 单测
- 压力模式下字段不饥饿单测
- 生命周期烟测回归

### Phase 2：把内部语义彻底收紧

目标：

- 让系统真正落在“单生命周期 + 事实缓存 + 判据评估”模型上

#### Commit B1：收缩 `AxisReadinessEvaluator`

改动目标：

- 明确它只输出判据，不继续向轴级状态机扩张

#### Commit B2：收缩 `SharedDriverState`

改动目标：

- 把调度内部状态彻底挡在共享状态面之外

#### Commit B3：分离 per-device / per-profile refresh 策略

改动目标：

- 把 startup、steady、recover、pressure 的刷新策略从全局 knob 收拢成 per-device profile

#### Commit B4：文档与接口语义回收

改动目标：

- 更新架构文档、使用说明与测试文档
- 清理已经落后于现实实现的注释与说明

当前结果：

- `hw smoke` 已切换到 `rostest`
- startup probe 与 steady refresh 已明确区分为“per-device override + global default”
- 第二阶段后的内部模型已经收敛到“单生命周期 + 共享事实缓存 + 判据评估 + per-device refresh runtime”

## 6. 第一阶段的明确非目标

本轮不做以下事情：

- 不重写 `OperationalCoordinator` 的对外状态集合
- 不改生命周期 service 名
- 不重构 `motor_command` 维护接口语义
- 不动 MT 协议调度，除非为通用基础设施做最小配套修改
- 不把 PP 私有协议包装成仿 CiA402 状态机

## 7. 完成标准

当第一阶段完成时，应至少满足：

- PP 任一关键反馈字段最大调度间隔可解释、可验证
- lifecycle 健康判据不再被轮盘调度误导
- warning 能明确区分“字段未调度”和“字段调度后无响应”
- 多轴 steady 运行下不会再出现长秒级假 stale 告警

当第二阶段完成时，应满足：

- `can_driver` 对外生命周期契约继续与 `Eyou_Canopen_Master` 一致
- 内部只剩一个系统生命周期状态机
- 轴级只保留事实与判据，不再演化成第二套状态机
