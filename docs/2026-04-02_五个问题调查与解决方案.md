## 问题 1：`csp` 轴仍未完全纳入位置语意约束

### 现状复核

- 启动同步时，`csp` 已经和 `position` 一样对齐到当前反馈位置：
  - `src/CanDriverHW.cpp:486`
- 但启动越界检查仍只对 `position` 做：
  - `src/CanDriverHW.cpp:490`
- 运行时 `joint_limits` 钳位仍只对 `position` 生效：
  - `include/can_driver/CanDriverIoRuntime.h:521`
- `max_position_step_rad` 也仍只对 `position` 生效：
  - `include/can_driver/CanDriverIoRuntime.h:182`
- `set_zero_limit` 服务目前只允许 `position`，`csp` 被拒绝：
  - `src/CanDriverHW.cpp:1239`

### 根因

当前代码实际上默认了三套互相独立的模式语意：

- `velocity`
- `position`
- `csp`

但从业务语意上，`csp` 不该是一种“完全独立的第三类轴”，它更接近“位置目标语意 + 不同协议路由”。

现在的问题是：

- “目标量”这件事上，`csp` 被当成位置轴；
- “安全约束”这件事上，`csp` 又被当成非位置轴；
- “发送路由”这件事上，`csp` 再被单独分支处理。

这导致位置语意在系统里被切碎了。

### 风险

- `csp` 运行中可能绕过位置限位。
- 上电同步时 `csp` 超限不会被同等级拦截。
- 调零/软限位类能力不能一致覆盖 `csp`。

### 解决方案

不要到处写 `controlMode == "position"`，而是引入统一语意函数，例如：

- `usesPositionTarget(mode)`
- `usesPositionLimits(mode)`
- `usesVelocityTarget(mode)`
- `dispatchRoute(mode)`

最低可行改法：

1. 抽一个小 helper，先不大改架构：
   - `bool modeUsesPositionSemantics(mode)` 返回 `position/csp=true`
   - `bool modeUsesVelocitySemantics(mode)` 返回 `velocity=true`

2. 统一替换这些位置：
   - startup out-of-range check
   - `clampWithJointLimits()`
   - `max_position_step_rad`
   - `set_zero_limit`

3. 新增测试：
   - `csp` 启动越界应报错
   - `csp` 命令应被 position limits 钳位
   - `csp` 应受 `max_position_step_rad` 限制
   - `csp` 应允许 `set_zero_limit`

### 建议 commit

- `safety(csp): align csp with position semantic guards`

## 问题 2：写链路仍存在 prepare/dispatch 竞态

### 现状复核

- Prepare 阶段在 joint mutex 保护下读取 `joint.controlMode / posCmd / velCmd / directCmd / requireCommandAlignment`：
  - `include/can_driver/CanDriverIoRuntime.h:101`
- Dispatch 阶段又直接从活的 `(*joints)[index]` 读取 `joint.controlMode / joint.motorId / joint.name` 决定如何下发：
  - `include/can_driver/CanDriverIoRuntime.h:295`
  - `include/can_driver/CanDriverIoRuntime.h:349`
- 另一边，这些 joint 字段会在多个回调/服务路径被修改：
  - direct cmd callback：`src/CanDriverHW.cpp:734`
  - lifecycle hold：`src/CanDriverHW.cpp:777`
  - `CMD_SET_MODE`：`src/CanDriverHW.cpp:1162`

### 根因

当前系统把“一拍命令”的语意拆成了两阶段：

- Prepare：算命令值
- Dispatch：决定发到哪个协议 API

但这两阶段之间没有共享同一个不可变快照。

于是 prepare 时看到的是一种 mode，dispatch 时看到的可能已经是另一种 mode。

### 风险

- 模式切换瞬间出现“算的是位置命令，发的是速度路由”或反过来。
- 生命周期切换、恢复、UI 直控和主循环并发时，行为依赖时序。
- 这类问题在单元测试中不一定稳定复现，但实机会表现成偶发错路由、首拍异常或命令被吞。

### 解决方案

把 prepare 阶段的产物从“裸 raw buffer”升级成“不可变命令快照”。

建议新增结构，例如：

- `PreparedAxisCommand`
  - `valid`
  - `jointIndex`
  - `jointName`
  - `motorId`
  - `canDevice`
  - `route` (`Position` / `Velocity` / `Csp`)
  - `rawCommand`
  - `requiresEnabledCheck`
  - `requiresFaultCheck`

然后：

1. Prepare 阶段在 joint mutex 下完成：
   - 命令源选择
   - 限位
   - 对齐门控
   - step limit
   - 路由决策
   - `motorId/name/device` 固化

2. Dispatch 阶段只消费 `PreparedAxisCommand`
   - 不再去读活的 `joint.controlMode`
   - 最多只读取共享反馈快照和 fault/enable 结果

3. 如果 `stopIssuedOnFault` 仍需回写 joint，可单独保留一个细粒度更新点，但不要让它影响路由选择。

### 建议测试

- 构造 prepare 后、dispatch 前切模式的场景，验证仍按 prepare 时快照路由。
- 构造 lifecycle hold 与 direct topic 并发场景，验证命令不会交叉错发。

### 建议 commit

- `write_path: dispatch from immutable prepared snapshots`

## 问题 3：模式/命令语意仍没有单一真源

### 现状复核

模式语意目前仍分散在多处：

- joint 配置层保留字符串 `controlMode`
  - `include/can_driver/CanDriverHwTypes.h:167`
- `command_gate` 里自己分 velocity / non-velocity
  - `src/command_gate.cpp:12`
- `captureCommandSnapshots()` 再做一次目标值/实际值判断
  - `src/CanDriverHW.cpp:794`
- `PrepareCommands()` 再按 mode 分命令源和限位
  - `include/can_driver/CanDriverIoRuntime.h:109`
- `DispatchPreparedCommands()` 再按 mode 分协议路由
  - `include/can_driver/CanDriverIoRuntime.h:349`
- `CMD_SET_MODE` 还自带一套 mode decode + preload 逻辑
  - `src/CanDriverHW.cpp:1098`

### 根因

系统里没有一个“轴语意对象”来统一回答这些问题：

- 这个 mode 用位置目标还是速度目标？
- 当前实际值应该比较 `pos` 还是 `vel`？
- 对齐门控该用什么 tolerance？
- 应该走 `setPosition`、`setVelocity` 还是 `quickSetPosition`？
- 切模式后应该 preload 什么？
- 外发 `motor_states.mode` 应该映射成什么？

这些问题现在分别散落在：

- config 解析
- lifecycle gate
- write path
- mode switch service
- UI/状态发布

### 风险

- 新规则必须多点同步修改，极易漏改。
- 代码越来越“能跑，但解释不清”。
- 未来对齐 `Eyou_Canopen_Master` 时，胶水层会发现内部没有稳定外观。

### 解决方案

建议做一次明确的“轴命令语意收口”。

### 推荐设计

引入两个层次：

1. `AxisControlMode` 枚举
   - `Position`
   - `Velocity`
   - `Csp`
   - `Unknown`

2. `AxisCommandSemantics` helper
   - `usesPositionTarget(mode)`
   - `actualValue(joint, mode)`
   - `targetValue(joint, mode)`
   - `alignmentTolerance(joint, mode)`
   - `applyLimits(joint, mode, value)`
   - `applyStepLimit(joint, mode, value, maxStep)`
   - `dispatchRoute(mode)`
   - `preloadAction(mode, currentFeedback)`
   - `motorStateMode(mode)`

这样可以把下面这些地方统一改成“调用语意 helper”，不再各自写字符串判断：

- `command_gate`
- `captureCommandSnapshots()`
- `PrepareCommands()`
- `DispatchPreparedCommands()`
- `CMD_SET_MODE`
- `BuildMotorStateMessages()`

### 落地建议

为了避免一次性大改过猛，可以分两步：

1. 先保留 `controlMode` 字符串，对外兼容；内部新增 `AxisControlMode` 和 helper。
2. 再逐步把内部判断迁移到枚举和 helper，最后只在参数解析/日志/ROS 接口边界保留字符串。

### 建议 commit

- `axis_semantics: centralize mode and command meaning`

## 问题 4：`motor_states` 仍不是纯反馈态

### 现状复核

- `mode` 字段在 shared feedback 无效时，会回退到配置侧 `controlMode`：
  - `include/can_driver/CanDriverIoRuntime.h:457`
  - `include/can_driver/CanDriverIoRuntime.h:486`
- `enabled/fault` 只有 `statusValid` 时才填，否则保留默认值：
  - `include/can_driver/CanDriverIoRuntime.h:460`
- `MotorState.msg` 没有 validity/freshness/source 语义：
  - `msg/MotorState.msg:1`
- UI 直接把 `msg.enabled / msg.fault / msg.mode` 当真实状态显示：
  - `scripts/realtime_motor_ui.py:102`
  - `scripts/realtime_motor_ui.py:108`
  - `scripts/realtime_motor_ui.py:110`

### 根因

当前 `motor_states` 同时承担了两件事：

- 发布“电机真实反馈”
- 给上层一个“尽量别空着的状态展示”

这两个目标天然冲突。

一旦反馈缺失，代码就会倾向于：

- `mode` 用配置回退
- `enabled/fault` 用默认值
- 其他量继续发数值

对 UI 来说这很方便，但对状态机和胶水层来说，这会把“未知”伪装成“已知”。

### 风险

- UI 显示 enabled/mode 正常，但其实只是回退值。
- 胶水层把它当可靠状态输入，错误判断系统是否已就绪。
- 与 `Eyou_Canopen_Master` 做统一外观时，状态可信度不一致。

### 解决方案

建议把 `motor_states` 明确变成“反馈 + 有效性”接口，而不是“反馈或猜测”接口。

### 推荐方案 A：扩展现有消息

扩展 `MotorState.msg`，新增：

- `bool mode_valid`
- `bool status_valid`
- `bool position_valid`
- `bool velocity_valid`
- `bool current_valid`
- `bool feedback_fresh`

并修改发布规则：

- `mode` 无真实反馈时发布 `MODE_UNKNOWN`
- `enabled/fault` 无状态反馈时不再假装为 `false`，而是配合 `status_valid=false`
- UI 按 validity/freshness 显示“未知/陈旧”

### 备选方案 B：保持旧消息，再加一条新消息

例如新增 `MotorFeedbackState.msg`，专门给上层状态机/胶水层用。

优点：

- 不破坏现有 UI/topic 兼容性

缺点：

- 维护两套 topic，复杂度更高

### 建议

优先推荐方案 A，因为这个包本来就在重构期，直接把语意补全更干净。

### 建议测试

- shared feedback 缺失时，`mode` 应发布 `MODE_UNKNOWN`
- `enabled/fault` 缺失时应体现 `status_valid=false`
- UI 收到 invalid 状态时应显示“未知/陈旧”，而不是沿用旧值

### 建议 commit

- `motor_state: publish validity-aware feedback semantics`

## 问题 5：测试、服务定义、使用文档已和实现漂移

### 现状复核

- smoke 仍是单个 `rostest` 壳，并被 `time-limit=120.0` 卡住：
  - `tests/test_can_driver_hw_smoke.test:2`
- `MotorCommand.srv` 仍写 `CMD_SET_MODE value: 0=position 1=velocity`
  - `srv/MotorCommand.srv:9`
- 使用文档也仍写 `CMD_SET_MODE=3 (value: 0/1)`：
  - `docs/使用指南.md:99`
- 但实现已经支持 `2=csp`，而且模式切换后会做 preload + alignment gate：
  - `src/CanDriverHW.cpp:1098`
- 使用文档还写着 `enabled/fault` “当前版本未填充”，但代码现在在 `statusValid` 时已经会填：
  - `docs/使用指南.md:110`
  - `include/can_driver/CanDriverIoRuntime.h:460`

### 根因

最近几轮重构主要发生在实现层：

- 生命周期
- 模式切换
- CSP 路由
- shared feedback
- gate / preload

但对应的外部契约层没有同步更新：

- `.srv/.msg` 注释
- 使用指南
- smoke 测试壳和测试粒度

### 风险

- 联调时大家会按旧文档调用服务。
- 测试失败不一定代表代码坏，测试通过也不一定代表新语意被覆盖。
- 团队最终只能相信抓包和实机，不再相信测试与文档。

### 解决方案

这一项建议分成 4 个收口动作：

1. 修服务/消息注释
   - `MotorCommand.srv` 写清 `0=position, 1=velocity, 2=csp`
   - 如果问题 4 采用 validity 方案，同步更新 `MotorState.msg`

2. 修使用文档
   - 模式切换真实语意
   - preload 行为
   - alignment gate 行为
   - `motor_states` 当前哪些字段是真反馈，哪些受 validity 约束

3. 拆测试壳
   - 非 ROS graph 依赖的用例尽量回到普通 `gtest`
   - `rostest` 只保留少量真正需要 launch/service/topic 的烟雾路径
   - 大文件 `test_can_driver_hw_smoke.cpp` 进一步按主题拆分

4. 清理名义依赖
   - 例如 `actionlib/action` 当前在主链里没有明确消费点，后续可评估是否去掉生成和依赖

### 建议 commit

- `docs(contract): align services, topics and mode semantics`
- `test(smoke): split ros graph smoke from plain gtests`
- 可选：`build: trim unused action dependencies`

## 建议的总体落地顺序

建议按下面顺序处理：

1. `csp` 安全语意一致化
2. write path 原子快照化
3. 轴命令语意收口
4. `motor_states` validity 语意补全
5. 文档/测试/服务定义收口

原因：

- 1 和 2 会直接影响实机稳定性和安全性，优先级最高。
- 3 是防止继续“越修越散”的根因修复。
- 4 决定上层能否相信状态。
- 5 则把对外契约和 CI/联调环境重新对齐。

## 最后判断

这 5 个问题里，真正最深的不是第 5 点，而是第 3 点。

第 1、2、4、5 都可以看作第 3 点的外显症状：

- `csp` 安全规则不一致，是因为模式语意散了。
- dispatch 竞态，是因为一拍命令没有单一快照。
- `motor_states` 混反馈和猜测，是因为“状态语意”没有被明确建模。
- 文档和测试漂移，是因为外部契约没有单独被当成一层维护。

所以如果只修表面 bug，不先收口第 3 点，后面还会继续长出同类问题。
