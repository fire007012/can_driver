# 意优私有协议单一生命周期设计

日期：2026-03-31

## 1. 结论

基于《意优行星关节 CAN 通讯协议》，`can_driver` 更合适的目标架构不是
“生命周期状态机 + 仿 CiA402 轴级状态机”，而应该是：

- 一个唯一的系统生命周期状态机：`OperationalCoordinator`
- 一层轴级事实缓存：真实寄存器回读和链路信息
- 一层轴级判据评估：回答“为什么现在不能运行”

也就是说：

- 系统只有一套权威状态
- 轴级不再拥有第二套 `Armed/Running/Faulted/...` 命名状态
- 轴级只提供事实和准入判据

## 2. 协议真实能力

从本仓库内的手册 [`意优行星关节CAN通讯协议.md`](/home/dianhua/Robot24_catkin_ws/src/can_driver/docs/意优行星关节CAN通讯协议.md#L5) 可知，这类电机提供的是“寄存器式控制语义”，而不是 CiA402 那种完整设备状态机。

协议里和运行控制直接相关的核心寄存器只有：

- `0x0F`：工作模式。支持位置、速度、电流、`CSP` 等模式。
- `0x10`：使能/失能状态，`01=使能`，`00=失能`。
- `0x11`：结束当前运行，但电机不失能。
- `0x15`：告警指示。写 `0` 清告警，读值判断故障是否仍存在。
- `0x07 / 0x06 / 0x05`：位置、速度、电流反馈。

命令通道语义也很明确：

- `0x01` 普通写：需要 `0x02` 写返回确认。
- `0x03` 读：需要 `0x04` 读返回。
- `0x05` 快写：立即生效、无返回。

这意味着：

- 电机有“模式 / 使能 / 故障 / 运动反馈”这些事实
- 但没有 CiA402 `statusword` 那种完整标准状态图
- 主机不应在驱动层再额外虚构一套与系统生命周期平行的轴状态机

## 3. 为什么不能直接照搬 `Eyou_Canopen_Master`

`Eyou_Canopen_Master` 采用双层结构是合理的，因为它的下层是真实 CiA402 设备状态机：

- 上层 `OperationalCoordinator` 负责系统操作模式与策略
- 下层 `CiA402Protocol`/设备反馈负责标准状态字解码和控制字生成

而 `can_driver` 面对的是私有协议电机：

- 没有标准 `statusword`
- 没有 `ReadyToSwitchOn / SwitchedOn / OperationEnabled` 这类原生设备态
- 只有模式、使能、告警、位置/速度/电流这些寄存器事实

所以 `can_driver` 正确的借鉴方式应该是：

- 借鉴 `Eyou_Canopen_Master` 的“单一权威系统状态”
- 不借鉴它的“设备原生状态机层”

## 4. 建议的目标架构

建议把 `can_driver` 收敛到下面这条主链：

`ServiceGateway -> OperationalCoordinator -> LifecycleDriverOps -> DeviceManager/EyouCan -> SharedDriverState`

各层职责建议如下：

- `OperationalCoordinator`
  只维护唯一系统状态，处理 `init / enable / release(对外为 resume) / halt / disable / recover / shutdown`。
- `LifecycleDriverOps`
  负责把系统动作映射为协议动作，并聚合所有轴的准入条件。
- `EyouCan`
  只做协议编码、发送调度、ACK/读回填、寄存器缓存。
- `SharedDriverState`
  只保存轴事实与设备事实，不再保存第二套轴级生命周期状态。
- `AxisRuntime`
  应重命名并收缩为“轴级判据评估器”，例如 `AxisReadinessEvaluator`。

## 4.1 对外生命周期契约对齐原则

虽然 `can_driver` 的内部实现不应继续仿造 CiA402 轴级状态机，但它的对外生命周期契约应与
`Eyou_Canopen_Master` 保持一致，便于后续通过一层胶水层做统一接入。

对外必须保持一致的内容包括：

- 相同的 service 名：`init / enable / disable / halt / resume / recover / shutdown`
- 相同的系统状态名：`Configured / Standby / Armed / Running / Faulted / Recovering / ShuttingDown`
- 相同的成功态语义：
  - `init` 成功后进入 `Armed`
  - `enable` 成功后进入 `Armed`
  - `resume` 成功后进入 `Running`
  - `halt` 成功后回到 `Armed`
  - `disable` 成功后回到 `Standby`
  - `recover` 成功后回到 `Standby`

这意味着：

- `can_driver` 可以内部使用“事实层 + 判据层”
- 但这些内部判据不能泄露成另一套外部生命周期语义
- 胶水层不应关心电机是不是私有协议，只应看到与 `Eyou_Canopen_Master` 同构的系统生命周期

## 5. 唯一系统状态机

建议系统状态只保留这一套：

- `Configured`
- `Standby`
- `Armed`
- `Running`
- `Faulted`
- `Recovering`
- `ShuttingDown`

状态语义建议固定为：

- `Configured`：驱动已启动，但未建立可运行设备会话。
- `Standby`：设备在线、无故障、处于失能态。
- `Armed`：设备已使能、模式正确，但主机侧仍冻结运动输出。
- `Running`：允许下发正常位置/速度/CSP 命令。
- `Faulted`：任一轴故障或关键链路失活。
- `Recovering`：正在执行故障清除确认流程。
- `ShuttingDown`：正在下线。

这里最关键的一点是：

- `Armed` 和 `Running` 是系统层含义，不是设备层含义
- 电机协议只提供“使能了没、模式对没、报警清没”

## 6. 轴级只保留事实

每轴建议只维护以下事实字段：

- `feedback_seen`
- `feedback_fresh`
- `online`
- `mode_actual`
- `mode_desired`
- `enabled_actual`
- `alarm_code`
- `position`
- `velocity`
- `current`
- `last_rx_ns`
- `last_mode_match_ns`
- `last_enable_match_ns`
- `last_alarm_clear_ns`
- `last_config_ack_addr`
- `last_config_ack_state`

必要时还可补：

- `last_halt_command_ns`
- `last_recover_command_ns`
- `last_enable_command_ns`

但这些都是“事实记录”，不是状态名。

## 7. 轴级只输出判据

轴级评估器只需要回答下面这些问题：

- `device_ready`
- `feedback_ready`
- `mode_ready`
- `enabled_ready`
- `fault_cleared`
- `recover_confirmed`
- `axis_ready_for_enable`
- `axis_ready_for_run`
- `block_reason`

例如：

- `mode_ready = desired_mode_valid && mode_actual == mode_desired`
- `enabled_ready = enabled_actual == true`
- `fault_cleared = alarm_code == 0`
- `feedback_ready = feedback_seen && feedback_fresh && online`
- `axis_ready_for_run = feedback_ready && mode_ready && enabled_ready && fault_cleared`

这层不再返回：

- `Armed`
- `Running`
- `Faulted`
- `Recovering`

因为这些名字应该只属于系统生命周期。

## 8. 协议动作到生命周期动作的映射

### 8.1 `Init`

目标：

- 建立 CAN 设备和协议对象
- 启动反馈轮询
- 读取基础寄存器，拿到第一批事实
- 对纳入管理的轴完成进入冻结待运行态所需的准备动作

建议语义：

- 对外成功语义必须与 `Eyou_Canopen_Master` 一致：`Configured -> Armed`
- 成功条件不应只是“CAN 打开了”，而应是所有纳入管理的轴都具备：
  `feedback_ready && mode_ready && enabled_ready && fault_cleared`
- 主机侧命令闸门在 `init` 成功后仍保持冻结，因此 `Armed` 不等于 `Running`
- 若发现 `alarm_code != 0`，应进入 `Faulted`
- 若基础通信建立不完整或首批反馈不足，`init` 应失败，且不得假装进入 `Armed`

### 8.2 `Enable`

协议动作：

- 写 `0x10 = 1`
- 优先等待 ACK
- 之后加速轮询 `0x10 / 0x15 / 0x0F`

状态语义：

- `Standby -> Armed`
- 成功条件：`enabled_actual == true && alarm_code == 0`
- 该动作主要用于 `disable` 或 `recover` 之后重新回到冻结待运行态

### 8.3 `Release`（对外 service 名仍为 `resume`）

协议动作：

- 不额外发“进入运行”命令
- 只打开主机侧命令闸门

状态语义：

- `Armed -> Running`
- 成功条件：所有轴 `axis_ready_for_run == true`

这点很重要：

- `Release` 是主机行为，不是设备状态切换命令

### 8.4 `Halt`

协议动作：

- 写 `0x11 = 1`

状态语义：

- `Running -> Armed`

因为手册已经明确：

- `0x11` 只是结束当前运行，不会失能

### 8.5 `Disable`

协议动作：

- 写 `0x10 = 0`
- 读回确认 `enabled_actual == false`

状态语义：

- `Running/Armed/Standby -> Standby`

### 8.6 `Recover`

协议动作：

- 写 `0x15 = 0`
- 在短时间窗口内提高 `0x15 / 0x10 / 0x0F` 轮询频率
- 直到确认 `alarm_code == 0`

状态语义：

- `Faulted -> Recovering -> Standby`

成功条件应以回读为准，不应只看写命令是否发出。

这里也要与 `Eyou_Canopen_Master` 保持一致：

- `recover` 只负责清故障并回到 `Standby`
- `recover` 成功后不得隐式 `enable`
- `recover` 成功后不得隐式 `resume`
- 上层若要恢复运动，调用顺序仍应是 `recover -> enable -> resume`

## 9. 控制面与数据面分离

基于手册，建议明确分成两条链：

### 控制面

用于：

- `setMode`
- `Enable`
- `Disable`
- `Halt`
- `Recover`

特点：

- 低频
- 需要 ACK 或读回确认
- 应进入 `Config/Recover/Control` 类队列

### 数据面

用于：

- `CSP` 下 `quickSetPosition`
- 位置/速度控制连续流

特点：

- 高频
- 非阻塞
- 无 ACK
- 只保最新目标

这两条路径不应该混成一个“都算命令”的共享语义。

## 10. 查询策略建议

这类私有协议最怕的不是“没有 CiA402 状态机”，而是“关键事实刷得太慢”。

建议长期保持下面的原则：

- `position / velocity`：运行态高频刷新
- `mode / enabled / alarm`：模式切换、使能、恢复阶段高频刷新
- `current`：低于前两者，可在稳态降频
- 一旦出现 `txBackpressure`：
  先削减可选查询，再保控制帧和关键状态查询

也就是说：

- 优先保 `mode / enabled / alarm`
- 最先牺牲 `current`

## 11. 对现有 `can_driver` 的重构建议

### 第一阶段：语义收缩

- 保留 `OperationalCoordinator`
- 保留 `LifecycleDriverOps`
- 保留 `SharedDriverState`
- 将 `AxisRuntime` 改名为 `AxisReadinessEvaluator`

### 第二阶段：去状态机化

- 删除 `AxisRuntimeState`
- 删除 `Offline/Seen/Standby/Armed/Running/Faulted/Recovering` 这些轴级命名状态
- 改为返回 `AxisReadiness` 或等价结构

### 第三阶段：统一准入口

将以下判断全部收口到 `LifecycleDriverOps`：

- `motionHealthy()`
- `recoverAll()`
- `anyFaultActive()`

统一由“事实 + 判据”决定，不再借轴级状态名做推导。

## 12. 最终判断

基于协议手册，这类电机最合适的结构是：

- 一个系统生命周期状态机
- 一个简单的寄存器事实层
- 一个轴级准入判据层

而不是：

- 一个系统生命周期状态机
- 再叠一套仿 CiA402 的轴级生命周期状态机

一句话总结：

`Eyou_Canopen_Master` 是“生命周期 + 真实设备状态机”；  
`can_driver` 应该是“生命周期 + 事实缓存 + 判据评估”，不该继续向“简化版 CiA402”演化。

### 13.2 尚未完成的最大差异

当前与 `Eyou_Canopen_Master` 相比，剩余最大的架构差异不是生命周期命名，
而是**通信激活时机**：

- `CanDriverHW::init()` 期间仍会提前执行：
  - `startMotorRefreshThreads()`
  - `syncStartupPositionAndCommands()`
  - `applyInitialModes()`
  - `stateTimer_` 与 `active_` 激活
- 这意味着节点刚启动、尚未调用 `~/init` 时，驱动已经开始部分通信与状态刷新

这会带来两个后果：

- 对外虽然已经是 `Configured`，但内部并非真正“通信未激活”
- 若后续要与 `Eyou_Canopen_Master` 做统一胶水层，`~/init` 的职责边界仍然不够纯

### 13.3 下一步建议

若继续按本设计推进，下一步应优先处理“延迟激活到 `~/init`”：

- 将 `startMotorRefreshThreads()` 从 `CanDriverHW::init()` 迁出
- 将启动位置同步与 CSP 初始模式写入迁移到 lifecycle `init` 路径
- 让 `Configured` 真正表示“参数和对象已就绪，但总线运行时尚未进入工作态”
- 让 `~/shutdown` 后重新 `~/init` 与“冷启动后的首次 `~/init`”具有同构语义
